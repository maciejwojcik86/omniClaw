# Nullclaw Configuration Explanation

This document explains the settings found in your [agent_director_01/.nullclaw/config.json](file://wsl.localhost/Ubuntu/home/agent_director_01/.nullclaw/config.json) file. The settings are divided into sections, detailing what each variable controls and describing the available options.

## Global Defaults

*   `default_temperature`: (Number) Controls the randomness or creativity of the language model's responses. Lower values are more deterministic; higher values (e.g., 0.7) are more creative.

## Agents (`agents`)
Configures the language models and default properties for the agents.

*   `defaults.model.primary`: (String) Specifies the default AI model string for the agent to use (e.g., `"openai-codex/gpt-5.3-codex"`).
    *   *Options*: Typically mapped in the format `provider/model` (like `openrouter/anthropic/claude-sonnet-4` or `openai/gpt-4o`).

## Diagnostics (`diagnostics`)
Controls the logging and telemetry options for debugging and visibility.

*   `backend`: (String) The format/destination for diagnostic logs. (e.g., `"jsonl"`).
*   `log_tool_calls`: (Boolean) Whether to log whenever an agent invokes a tool or function.
*   `log_message_receipts`: (Boolean) Logs the receipt states of messages from various channels.
*   `log_message_payloads`: (Boolean) Logs the full payload of messages sent and received.
*   `log_llm_io`: (Boolean) Logs the raw input prompts and output responses from the Language Model.

## Autonomy (`autonomy`)
Defines the permissions and boundaries of the agent's execution abilities.

*   `level`: (String) The extent to which the agent can act independently. Set to `"supervised"` here. Other options typically include `"full"`.
*   `workspace_only`: (Boolean) Restricts the agent to read/write only within a designated workspace directory.
*   `max_actions_per_hour`: (Integer) Rate-limits the agent's operations.
*   `require_approval_for_medium_risk`: (Boolean) Whether actions flagged as "medium risk" halt for human approval.
*   `block_high_risk_commands`: (Boolean) Prevents the execution of dangerously destructive shell commands entirely.
*   `allowed_commands`: (List of Strings) A strict allowlist of shell commands the agent can execute (e.g., `"ls"`, `"cat"`, `"python3"`, etc.). You can use `"cmd *"` to allow sub-arguments, or `"*"` to allow everything (if full autonomy is enabled).
*   `allowed_paths`: (List of Strings) The absolute paths on the host system the agent is permitted to interact with.

### Current Limit Hotspots (Operational Review)
Even after setting `autonomy.allowed_commands` to `["*"]`, these fields can still limit runtime behavior:

*   `autonomy.level`: if still `"supervised"`, behavior remains policy-gated.
*   `autonomy.workspace_only`: if `true`, non-workspace path access can still be blocked.
*   `autonomy.allowed_paths`: if paths point to another user workspace, file operations can fail.
*   `autonomy.max_actions_per_hour`: low values can trigger `rate limit exceeded` during multi-step form handling.
*   `autonomy.block_high_risk_commands`: if `true`, high-risk shell actions remain blocked.
*   `agent.max_tool_iterations`: can truncate long multi-tool workflows.
*   `tools.shell_timeout_secs`: long-running commands can be killed early.
*   `tools.shell_max_output_bytes`: large command output can be truncated.

## Runtime (`runtime`)
Configures the execution environment for tasks and code generated by the agent.

*   `kind`: (String) The environment type. Set to `"native"` (running directly on the host) but can also be sandboxed via Docker.
*   `docker`: (Object) If enabled, specifies what Docker image (`"alpine:3.20"`), memory limit, and networking features (like `"network": "none"`) are used to sandbox code execution safely.

## Reliability & Scheduler (`reliability` & `scheduler`)
Configures retry logic and concurrency limits to maintain system stability.

*   `provider_retries` / `provider_backoff_ms`: (Integer) Controls how many times to retry API errors and how long to wait between attempts.
*   `channel_initial_backoff_secs` / `channel_max_backoff_secs`: Timeouts for connecting to messaging channels (like Telegram/Discord).
*   `scheduler.max_tasks` / `scheduler.max_concurrent`: Limits how many background tasks the agent orchestrator can handle simultaneously.

## Agent Behavior (`agent`)
Fine-tunes the context processing, tool behavior, and history limits.

*   `max_tool_iterations`: (Integer) The maximum number of tool steps the agent can take sequentially before stopping or asking for help.
*   `max_history_messages`: (Integer) Limits the number of past messages to keep in the immediate sliding-window context.
*   `parallel_tools`: (Boolean) Controls whether the agent can invoke multiple tools at the exact same time.
*   `compact_context`: (Boolean) Determines whether older messages should be aggressively summarized to save tokens.
*   `session_idle_timeout_secs`: (Integer) The time before a conversation session expires.

## Channels (`channels`)
Configures the messaging and user interface endpoints where the agent connects.

*   `telegram`: Your current configuration connects to Telegram via a `bot_token`.
    *   `allow_from`: `["*"]` implies anyone can talk to the bot, or you can restrict it to specific Telegram usernames.
    *   `reply_in_private`: Whether the bot restricts replies to Direct Messages.

## Memory (`memory`)
The subsystem responsible for long-term memory, conversation history, and Retrieval-Augmented Generation (RAG).

*   `profile` / `backend`: How memories are saved (e.g., `"markdown"`, `"sqlite"`, `"postgres"`).
*   `auto_save`: (Boolean) Automatically records memories from conversations.
*   `search.provider`: (String) Configures the embedding provider used for searching memory.
*   `search.model`: Configurations for vector embeddings (e.g., `"text-embedding-3-small"`).
*   `search.store.qdrant_url` / `pgvector_table`: Allows integration with vector databases like Qdrant or Postgres (PGVector) for scalable memory storage.
*   `lifecycle.archive_after_days` / `purge_after_days`: Automatic cleanup of old memories or history.

## Gateway (`gateway`)
The local API interface for nullclaw.

*   `port` / `host`: Binds the API server. (Default `127.0.0.1:3000`).
*   `require_pairing`: (Boolean) Forces external clients (like a Web UI) to authenticate or pair with the backend.
*   `webhook_rate_limit_per_minute`: Prevents spam on the gateway.

## Integrations (Browser, HttpRequest, Composio, Hardware)
Several optional components that expand the agent's capabilities:

*   `browser.enabled`: Controls whether the agent can spawn a headless browser to surf the web. Can connect to Chrome via `native_webdriver_url`.
*   `http_request.enabled`: Allows the agent to make raw HTTP requests. Often used in tandem with web search providers like `searxng` or `duckduckgo`.
*   `composio.enabled`: Connects to third-party integrations via the Composio platform.
*   `hardware.enabled`: Allows interaction with physical peripherals over serial ports (`baud_rate`, `serial_port`).
*   `tools.shell_timeout_secs` / `tools.max_file_size_bytes`: Limits on shell commands and file reading.

## Cost & Security (`cost`, `security`)

*   `cost.enabled`: You can define hard stop limits (`daily_limit_usd`) to ensure the agent doesn't overspend on API LLM tokens.
*   `security.sandbox.backend`: Restricts code execution.
*   `security.resources`: Imposes strict limits on Memory footprint (`max_memory_mb`) and CPU time (`max_cpu_time_seconds`) for isolated tasks.
*   `security.audit.enabled`: Logs all high-level security events to an `audit.log`.

---
*Note: This configuration maps directly to the OpenClaw / Nullclaw specification. Some features remain `null` or `false` by default to preserve a secure, minimal initialization footprint.*
